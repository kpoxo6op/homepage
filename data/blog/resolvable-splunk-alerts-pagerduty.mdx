I was impressed by this [post](https://community.splunk.com/t5/Random/Auto-Resolving-Pagerduty-Alerts/m-p/504687) on Splunk support forum explaining
how to automatically resolve PagerDuty alerts received from Splunk.

It took me some time to understand the post, so I decided to expand
it to an easy-to-follow guide to help others.

It's remarkable that it's been 4 years since the original post but there is still no support for autoresolving Splunk alerts as per PagerDuty [docs](https://www.pagerduty.com/docs/guides/splunk-integration-guide/)

This guide explains how to build the fully working demo.
Feel free to follow along step by step or just read the contents.

## Table of Contents

## Before you begin

Sign up for [Splunk](https://www.splunk.com/en_us/sign-up.html) and [PagerDuty](https://www.pagerduty.com/sign-up/) accounts.

Clone GitHub [repository]() with the demo code.

Install terraform, docker and docker compose.

## Set PagerDuty Token

Create [API Token](https://support.pagerduty.com/docs/api-access-keys#generate-a-user-token-rest-api-key) and save it to `.env` file.

```sh
cp .env.sample .env
cat .env
export TF_VAR_pagerduty_token='your_pagerduty_token'
```

## Configure Local Splunk

Start local docker compose Splunk stack, wait for healthy Splunk status.

```sh
docker compose up -d
watch docker compose ps
```

Login to local Splunk with credentials from `env` (default is `admin/abcd1234`).

Install PagerDuty [App](http://localhost:8000/en-US/manager/search/appsremote?query=pagerduty). I could not preload it to Splunk so we have to install it manually.

## Run the Demo

Run terraform to create the required resources in PagerDuty and Splunk.

```sh
source .env
terraform init
terraform plan
terraform apply
```

## How It Works

```text
+----------------------+    +-------------------+    +-------------------+    +-----------------+
| Splunk Alert Search  | -> | Check KVStore for | -> | PagerDuty Action  | -> | PagerDuty Event |
|  (Triggers/Resolves) |    | State Change      |    | Custom Details    |    | Orchestration   |
|                      |    |                   |    |                   |    |                 |
|                      |    +-------------------+    +-------------------+    +-----------------+
|                      |    | If State Changed  |    
|                      |    | Update KVStore    |    
+----------------------+    +-------------------+    
```

### Splunk Alert Search

Splunk Alert Search tracks alert state using the Splunk KV Store.

It decides whether to return results with `event_action` set to Trigger,
Resolve, or not return any results based on the current state.

```sql
// Randomly generate 0 or 100 search results to simulate intermittent errors
| makeresults count=1
| eval random_number=random() % 2
| appendpipe [| makeresults count=100 | where random_number == 1]
| where random_number == 1
| stats count as event_count
// trigger or resolve alert based on result count
| eval event_action=case(
    event_count>0, "trigger",
    1=1, "resolve"
  )
| eval _key="boris_as_code_error_rate"
| eval date_last_run=now()
// Check KV Store for the previous state and update if needed.
// KV Store maintains the alert state across searches to track state changes.
// This prevents duplicate alerts and ensures only state changes are sent to PagerDuty.
| join type=left _key
    [| inputlookup state_alert
    | rename event_action AS event_action_lookup
    | rename date_last_change AS date_last_change_lookup
    | fields _key, event_action_lookup, date_last_change_lookup]
// Save the updated state back to the KV Store.
| eval date_last_change=case(
    event_action!=event_action_lookup, now(),
    1=1, date_last_change_lookup
  )
| outputlookup state_alert append=true key_field=_key
// Return results only if the state has changed.  
| where event_action!=event_action_lookup
| table event_action
```

Trigger Action sends the alert event to PagerDuty with `event_action` in message `custom_details`.

### PagerDuty API

Example event received by PagerDuty API:

```json
{
	"custom_details": {
		"event_action": "trigger"
	},
	"dedup_key": "Resolvable Splunk Alert Demo",
	"event_action": "trigger",
	"severity": "unknown",
	"source": "unknown host",
	"summary": "Resolvable Splunk Alert Demo",
	"timestamp": "2024-05-27T21:08:00.917Z"
}
```

Example Event with `custom_details.event_action` set to Resolve:

```json
{
	"custom_details": {
		"event_action": "resolve"
	},
	"dedup_key": "Resolvable Splunk Alert Demo",
	"event_action": "trigger",
	"severity": "unknown",
	"source": "unknown host",
	"summary": "Resolvable Splunk Alert Demo",
	"timestamp": "2024-05-27T21:09:00.972Z"
}
```

Notice the difference between `event_action` and `custom_details.event_action`.

PagerDuty Splunk App creates `event_action`. It is always `trigger` because the app [does not support](https://www.pagerduty.com/docs/guides/splunk-integration-guide/) resolving alerts which are no logner firing.

The Splunk Alert Search creates `custom_details.event_action`. It can be `Resolve` or `Trigger`.

### PagerDuty Orchast

PagerDuty creates an incident because the `event_action` is set to `trigger.`

PagerDuty Event Orchestration rule is set to resovle  an incident if `custom_details.event_action` matches `resolve`.

See Event Orchestration rule:

```text
resource "pagerduty_event_orchestration_service" "resolve" {
  service = data.pagerduty_service.default.id
  set {
    id = "start"
    rule {
      label = "Autotoresolve Splunk Alerts"
      condition {
        expression = "event.custom_details.event_action matches 'resolve'"
      }
      actions {
        event_action = "resolve"
      }
    }
  }

  catch_all {
    actions {

    }
  }
}
```
